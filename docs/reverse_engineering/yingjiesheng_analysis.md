# 应届生求职网 - 抓包与反爬分析报告 (阶段 1)

## 1. 入口选型与优先级
- **Web端**: `http://my.yingjiesheng.com/` (或类似子域名)
- **特点**: 相较于其他头部平台，应届生求职网的整体反爬策略相对较弱，技术栈较为传统（大量基于服务端渲染的 HTML）。

**当前结论**: 依然首选 **Web 端**，通过 HTML 解析获取数据。

## 2. 核心接口分析 (Web端)

### 2.1 职位搜索列表页面
- **URL**: `http://my.yingjiesheng.com/index.php/personal/jobsearch` 或旧版的 `http://www.yingjiesheng.com/shanghai/`
- **Method**: GET
- **Query Params**: 依赖表单提交或 URL 路径传参（如按城市、关键词等）。
- **响应特征**: 直接返回 HTML，通过 DOM 解析即可提取职位列表。

### 2.2 反爬机制分析
1. **Cookie 时效性与 Session**:
   - 依赖于传统的 `PHPSESSID` 或类似的基础 Session Cookie。
   - 对未携带 Cookie 或 Cookie 过期的请求，可能重定向到登录页或返回提示。
2. **频率限制**:
   - 对单 IP 的高频请求有基于 Nginx/WAF 层面的限流，超过阈值会返回 403 Forbidden 或要求输入简单的图形验证码。
3. **User-Agent 校验**:
   - 存在基础的 UA 黑名单检测（如拦截默认的 `python-requests`）。

## 3. 后续突破方向

1. **会话与 Cookie 管理**:
   - **突破计划**: 设计一个简单的会话保持模块，首次请求获取初始 Cookie，并在后续请求中持续携带。一旦检测到响应状态码异常（如 302 重定向到登录，或 403），立即丢弃该会话，切换代理 IP 并重新初始化会话。
2. **基于 HTML 的容错解析**:
   - **突破计划**: 针对可能多变的 HTML 结构，编写基于 `BeautifulSoup` 或 `lxml` 的容错解析器，对于缺失的字段提供默认值处理，确保爬虫不会因个别页面的 DOM 变化而崩溃。
3. **分布式代理池控制**:
   - 核心在于“慢而稳”，在 `scheduler` 中配置该平台的请求间隔（如随机 2-5 秒），通过横向扩展代理 IP 数量来提升总体抓取速度。
