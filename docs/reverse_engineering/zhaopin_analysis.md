# 智联招聘 - 抓包与反爬分析报告 (阶段 1)

## 1. 入口选型与优先级
- **Web端**: `https://sou.zhaopin.com/`
  - 优点：支持完善的筛选条件，JSON API 数据格式清晰。
  - 缺点：核心 API 有严格的设备指纹绑定和签名校验（`x-zp-token` 及 `x-zp-client-id`）。
- **微信小程序**:
  - 优点：绕过了浏览器指纹的复杂校验。
  - 缺点：可能有微信特有的 `code` 校验。

**当前结论**：优先分析 **Web 端 API**。突破其动态 Token 是核心任务。

## 2. 核心接口分析 (Web端)

### 2.1 职位搜索列表接口
- **URL**: `https://fe-api.zhaopin.com/c/i/sou`
- **Method**: GET
- **Query Params**:
  - `pageSize`, `cityId`, `kw`, `kt`, `p`
- **核心反爬 Header**:
  - **`x-zp-client-id`**: 设备指纹 ID，通常由前端脚本在首次访问时生成或从服务器获取并持久化在 LocalStorage/Cookie 中。
  - **`x-zp-page-request-id`**: 页面级请求追踪 ID，保证请求顺序的合法性。
  - **`x-zp-token` / `sign`**: 动态签名。通常是通过对请求参数、时间戳、设备指纹等信息进行特定算法（如 AES 变种或自定义 Hash）加密生成的。缺失或错误会返回类似 `{"code": 403, "message": "签名错误"}`。

## 3. 疑似加密点与后续突破方向

1. **`x-zp-token` 逆向生成 (核心难点)**:
   - **定位**: 在 DevTools 中给 `x-zp-token` 下请求头断点。
   - **机制**: 该 Token 通常由一个复杂的 JS 加密库负责生成，该库会读取浏览器的 UserAgent、屏幕分辨率、硬件并发数等生成设备特征，然后混入时间戳和请求路径生成最终的 Token。
   - **突破计划**: 
     - 使用 AST 解析并剥离外层混淆（如 webpack 打包的代码）。
     - 找到加密主函数，提取并用 Python/Node 模拟执行。确保传入的参数（特别是伪造的浏览器环境参数）能够稳定通过后端的校验。

2. **请求频率与设备关联控制**:
   - **机制**: 智联对单一 `x-zp-client-id` 的高频访问有严格监控，超过阈值会拉起验证码或直接封禁该设备 ID。
   - **突破计划**: 在 `crypto-engine` 中不仅要提供签名生成，还要维护一个“虚拟设备指纹池”。每次分配任务时，绑定一个独立的 IP 和一组固定的设备特征（`client-id`），确保指纹的一致性，同时避免单一指纹请求过快。
