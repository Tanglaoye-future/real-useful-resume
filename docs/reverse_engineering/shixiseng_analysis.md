# 实习僧 - 抓包与反爬分析报告 (阶段 1)

## 1. 入口选型与优先级
- **Web端**: `https://www.shixiseng.com/interns`
  - 优点：无需登录即可查看大量职位，接口抓包相对容易。
  - 缺点：核心数据（如公司名、薪资、职位名）在 DOM 或 JSON 中进行了**字体反爬（自定义 WebFont）**，即前端显示的文字与源码中的 Unicode 编码不一致。
- **微信小程序**:
  - 优点：通常小程序接口的数据是干净的（没有字体反爬）。
  - 缺点：需要解包小程序获取 `app_id` 和签名密钥，可能存在请求参数的 `sign` 校验。

**当前结论**：优先分析 **Web 端 API**。因为字体反爬技术在业界已有成熟的破解方案（动态下载 woff 文件并映射 TTF 字形），相比逆向小程序签名，维护成本可能更低。若字体反爬变动过快，再切入小程序抓包。

## 2. 核心接口分析 (Web端)

### 2.1 职位搜索列表接口
- **URL**: `https://www.shixiseng.com/api/interns/search`
- **Method**: GET
- **Query Params**:
  - `k`: 关键词（如 `实习`）
  - `c`: 城市代码（上海 `310000`）
  - `p`: 页码
  - `s`: 排序方式
  - `sign`: 疑似动态签名（某些版本/路径下存在）
- **响应特征**:
  - 返回 JSON 格式数据。
  - **核心反爬**：JSON 数据中的关键中文字符（如薪资 `150-200/天` 中的数字，或者公司名中的常见汉字）被替换为特殊的 Unicode 字符（如 `&#xee4a;`）。

### 2.2 职位详情接口
- **URL**: `https://www.shixiseng.com/api/intern/detail`
- **Query Params**:
  - `uuid`: 职位唯一标识（从列表接口获取）
- **响应特征**:
  - 同样存在字体反爬，`job_detail`（岗位职责/要求）字段包含大量自定义 Unicode 字符。

## 3. 疑似加密点与后续突破方向

1. **字体反爬破解 (核心难点)**:
   - **机制**: 每次请求或每天，服务器会动态生成一个 `.woff` 字体文件，通过 `@font-face` 引入。页面的特定 Unicode 字符依赖该字体文件映射到真实的字形（如把 `&#xee4a;` 渲染成 `2`）。
   - **定位**: 在 Network 的 `Font` 标签页，抓取类似 `https://www.shixiseng.com/interns/iconfonts/file?rand=xxx.woff` 的请求。
   - **突破计划**: 
     - 阶段2中，编写 Python 脚本，动态下载该 `.woff` 文件。
     - 使用 `fontTools` 库解析 TTF 字形（Glyph），对比已知的基础字形库（或通过 OCR 识别生成的字形图像），建立动态的 `Unicode -> 真实字符` 映射字典。
     - 在解析接口 JSON 前，先用字典替换所有的乱码字符。

2. **接口 `sign` 参数 (若启用)**:
   - 实习僧的 API 在不同风控等级下可能会开启签名校验。
   - **定位**: 在 JS 文件中搜索 `sign:` 或拦截请求。
   - **突破计划**: 通常是简单的 MD5(参数 + 盐) 或 Base64 变种。用 DevTools 下断点找出盐值，用 Go/Python 1:1 复现。

3. **频率控制**:
   - 实习僧对单 IP 的请求频率有一定限制，但不及 BOSS 直聘严格。
   - **突破计划**: 设置 `scheduler` 保证单线程间隔 2-3 秒，配合常规代理池即可绕过。
