# 前程无忧 (51job) - 抓包与反爬分析报告 (阶段 1)

## 1. 入口选型与优先级
- **Web端**: `https://we.51job.com/`
  - 优点：可以直接获取到内联 JSON 数据。
  - 缺点：使用了阿里云盾（或类似的安全防护产品），有极强的动态 JS 挑战机制。
- **APP/小程序**:
  - 优点：可能没有复杂的 Web 端安全盾挑战。

**当前结论**：优先尝试 **Web 端**，核心在于解决安全盾的 `acw_sc__v2` 挑战和滑块验证。

## 2. 核心接口分析 (Web端)

### 2.1 职位搜索页面 (服务端直出/内联JSON)
- **URL**: `https://we.51job.com/pc/search`
- **Method**: GET
- **Query Params**:
  - `keyword`: 搜索词
  - `jobArea`: 城市代码
  - `pageCode`: 页码
- **响应特征**:
  - 正常情况下，页面 HTML 中包含一个 `<script>` 标签，里面有 `window.__SEARCH_RESULT__ = {...}`，包含当前页的所有职位数据。
  - **核心反爬拦截**: 如果请求不带有效的 `acw_sc__v2` Cookie，或者触发了风控，服务器会返回一段混淆的 JS 代码（安全盾），而不是正常的 HTML。这段 JS 的作用是在浏览器中执行一段加密运算，生成一个叫做 `acw_sc__v2` 的 Cookie，然后自动刷新页面（HTTP 302/刷新指令）。

## 3. 疑似加密点与后续突破方向

1. **安全盾 (`acw_sc__v2`) 逆向 (核心难点)**:
   - **机制**: 当发起无状态请求时，服务器不返回真实内容，而是返回一段压缩混淆的 JS（形如 `var _0x...`），该脚本会计算一个特定的哈希值并设置到 Cookie `acw_sc__v2` 中，然后重载页面。如果这个 Cookie 值不对，请求会被永久拦截或拉起极验滑块。
   - **突破计划**: 
     - 方案 A (纯逆向): 提取返回的那段 JS 代码，使用 AST 进行还原，分析出其生成 `acw_sc__v2` 的逻辑（通常是一个固定的算法，传入动态的参数，如 `arg1`）。用 Python 脚本模拟这个计算过程，直接生成合法的 Cookie。
     - 方案 B (模拟执行): 使用基于 Playwright/Puppeteer 的环境，先访问一次页面，让浏览器自动完成 JS 挑战并种下 Cookie，然后将这个 Cookie 提取出来供并发爬虫使用。由于需要规模化，方案 A 是更优的工程化选择。

2. **滑块验证码绕过**:
   - **机制**: 在高频请求或 IP 评分较低时，会直接弹出滑块验证码。
   - **突破计划**: 
     - 采用图像识别（如 OpenCV/PaddleOCR 的边缘检测）计算滑块缺口距离。
     - 利用贝塞尔曲线算法生成拟人化的滑动轨迹。
     - 逆向验证接口，生成包含轨迹数据的加密 payload，或者直接在自动化浏览器中完成滑动后接管会话。
